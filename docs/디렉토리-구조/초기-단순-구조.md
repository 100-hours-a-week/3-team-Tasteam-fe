# 초기 단순 구조 (작게 시작)

프로젝트 규모가 작거나 팀이 적을 때는 폴더를 과하게 쪼개기보다
**라우팅 중심 + 공통 폴더** 정도로 시작하는 것이 관리가 쉽습니다.

## 폴더 구조 예시

```
src/
  app/         # 엔트리, 라우팅, 전역 설정
  pages/       # 라우팅 단위 화면
  components/  # 공통 UI 컴포넌트
  services/    # API 요청, 클라이언트
  lib/         # 유틸 함수
  styles/      # 전역 스타일
  assets/      # 이미지/폰트
  types/       # 공통 타입
```

## 장점

- 구조가 단순해 초기 진입장벽이 낮음
- 파일 이동/리팩토링 부담이 적음

## 단점

- 규모가 커지면 `components/services`가 빠르게 비대해질 수 있음
- 도메인/기능 단위 분리가 느려질 수 있음

## 언제 FSD로 확장할까?

- 페이지/기능이 10개 이상으로 늘어남
- 특정 도메인(예: user/team) 관련 파일이 한 폴더에 과밀해짐
- 신규 팀원이 구조를 이해하기 어려워지는 시점

## 전환 방법(요약)

- `components` → `shared/ui`
- `services` → `shared/api` + `entities/*/api`
- 페이지 내 로직 중 “기능”은 `features/`로 이동
- 도메인 타입/헬퍼는 `entities/`로 분리
