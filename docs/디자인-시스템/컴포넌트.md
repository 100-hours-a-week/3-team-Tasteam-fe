# 컴포넌트 가이드

이 문서는 `src/shared/ui/`에 추가되는 공용 컴포넌트의 설계/구현 기준을 정리합니다.

## 0) shadcn/ui 기반 운영

기본 컴포넌트는 shadcn/ui를 우선 채택하고, 우리 서비스 요구사항에 맞게 커스텀합니다.
이때 "원본에 가까운 shadcn 컴포넌트"와 "서비스 표준 API 컴포넌트"를 분리해 운영하는 것을 권장합니다.

자세한 결정/배치 전략은 `docs/디자인-시스템/결정사항.md`를 참고합니다.

## 1) 컴포넌트 분류(권장)

- Primitives(프리미티브): 버튼/텍스트/아이콘/스택(레이아웃)/입력 등
- Composed(조합): 프리미티브를 조합한 패턴(예: 검색바, 필터 바)
- Domain UI(도메인 결합): `features/`, `widgets/`로 이동(공용 UI 아님)

원칙:

- `shared/ui`에는 "도메인 의미"가 들어가지 않도록 합니다.
  - 예: `TeamCard`는 도메인 UI일 가능성이 큼 → `entities/team/ui` 또는 `widgets/`

## 2) API 설계 규칙(권장)

- props 네이밍은 일관되게 유지
  - 예: `variant`, `size`, `isLoading`, `isDisabled`
- 상태는 Boolean props로 표현(필요 시)
  - `disabled`는 HTML 기본 속성을 우선 사용
- 기본값을 제공해 사용자가 최소 props로 사용 가능하게
- "너무 많은 variant"는 지양(디자인 시스템이 흔들리는 신호)

## 3) 상태(state) 정의(필수)

모든 인터랙티브 컴포넌트는 최소 아래 상태를 정의합니다.

- `default`
- `hover` (pointer 환경)
- `active`
- `focus-visible` (키보드 포커스)
- `disabled`
- `loading` (해당 컴포넌트에 의미가 있을 때)
- `error` (입력 컴포넌트에 해당)

## 4) 접근성(필수)

- 클릭 가능한 요소는 기본적으로 `button`, `a` 등 올바른 시맨틱 태그를 사용
- 키보드 포커스는 `:focus-visible` 기반으로 제공
- 아이콘 버튼은 `aria-label` 필수
- 입력 요소는 `label`/`aria-describedby`로 도움말/에러 메시지 연결

## 5) 스타일링/토큰 사용(권장)

- 컴포넌트의 색/간격/반경은 토큰을 통해 제어합니다.
  - hard-coded HEX/px 남발을 피합니다.
- Tailwind를 쓰는 경우에도 값은 토큰(CSS 변수)에 연결합니다.
- Emotion을 쓰는 경우에도 값의 원천은 토큰(CSS 변수)로 둡니다.

## 6) 구현 체크리스트(추가 시 필수 확인)

- 컴포넌트 목적/사용 시나리오가 명확한가?
- variant/size/state가 문서화됐는가?
- 키보드 조작/포커스가 정상 동작하는가?
- 비정상 케이스(긴 텍스트, 작은 화면, 다국어)에 안전한가?
- `shared/ui`에 두는 것이 맞는가(도메인 결합 아님)?
