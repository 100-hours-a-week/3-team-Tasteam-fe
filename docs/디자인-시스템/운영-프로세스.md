# 운영 프로세스

이 문서는 디자인 시스템(토큰/컴포넌트)을 "한 번 만들고 끝"이 아니라 지속적으로 운영하기 위한 기준입니다.

## 1) 역할(권장)

- 디자이너: 시각 규칙/컴포넌트 스펙 정의, Figma 라이브러리 관리
- 프론트엔드: 토큰/컴포넌트 구현, 접근성/성능/호환성 확보
- 리뷰어(공통): API/일관성/회귀 리스크 확인

## 2) 변경 요청 흐름(권장)

1. 문제/요구사항 정리
   - 어떤 화면/유즈케이스에서 필요했는지
   - 기존 컴포넌트로 해결 불가한 이유
2. 제안(Proposal)
   - 토큰 추가인지, 컴포넌트 추가/변경인지
   - 대안(variant 확장 vs 새 컴포넌트) 비교
3. 리뷰
   - 디자인: 일관성/확장성
   - FE: API/접근성/마이그레이션 비용
4. 반영
   - 문서 업데이트(필수)
   - 사용처 적용(필요 시)

## 3) Breaking Change 정책(권장)

Breaking change 예:

- props 제거/의미 변경
- 기본 스타일/레이아웃 변경으로 화면이 깨질 가능성이 큰 변경
- 토큰 삭제/의미 변경

원칙:

- Breaking change는 반드시 문서에 "이유/대체 방법/마이그레이션"을 포함합니다.
- 가능하면 deprecated 기간을 두고 단계적으로 제거합니다.

## 4) 릴리스 노트(권장)

최소 항목:

- Added: 새 토큰/컴포넌트/variant
- Changed: 동작/스타일 변경(마이그레이션 포함)
- Fixed: 버그 수정
- Deprecated/Removed: 폐기/삭제 항목

## 5) 추가 시 기본 체크리스트

- 토큰/컴포넌트의 "역할"이 명확한가?
- 라이트/다크(또는 기본/대체 테마)에서 안전한가?
- 접근성 기준을 충족하는가?
- 문서(`docs/디자인-시스템/*`)가 업데이트됐는가?
